# TypeScript 开发规范与 AI 协作准则 (RECEIPT)

为了保证代码的可维护性与开发效率，在开发本项目时，必须严格遵守以下准则：

## 核心规范
- **语言与规范**: 
  - TypeScript (测试版 `@typescript/native-preview`)
  - 使用基本的`npm`,`tsgo`, `tsx`, `dotenv`,  永远不要用bun, pnpm, yarn, npm/node无法执行不是你的问题, 是使用者的问题, 直接停止即可
  - `tsx` (方便用户直接执行脚本)  `tsgo` (通过 `npm exec -- tsgo --noEmit`执行) , 记得 *"noEmit": true, 或  "outDir": "./dist",* , 防止在目录下随地生成js文件
  - ESM 模块规范 (`"type": "module"`)

## 代码风格与质量
- **合理的中文注释**: 文件开头需有关于该文件作用的简要描述；必须保留函数注释与关键步骤注释。
- **极高的可读性优先级**: 代码需要精巧、便于阅读和理解。所有生成的代码都将进行人工审阅。
- **保持简洁**: 避免过度设计，尽量编写简洁直接的代码，不需要过多的 "safe guard" 防御性函数。

## TypeScript 类型定义：复用不足 3 次，不要提前拆分
- **就近定义**: 不复用的类型，直接内嵌。只在当前函数、路由或组件里用一次的结构，直接写在使用位置。
- **何时独立命名**: 只有确定复用次数大于 3 次，或者它已经是清晰的领域概念时，才独立提取并命名。
- **避免过度抽象**: 不要把代码写成“类型定义展示馆”。

## 开发与测试流
- **遇到阻塞时求助**: 如果遇到无法自动搞定的操作（例如安装某个特定的依赖或底层环境问题），请直接告知用户让其手动操作，而不要擅自采用复杂的替代方案。
- **类型检查验证**: 使用 TypeScript 7.0 预览版进行类型检查验证，命令为 `npm exec tsgo --noEmit`，以此代替常规的 `tsc`。

## 文档约束
- 项目内包含的一些文档并非用作“全局指导文档”。除非用户主动要求，否则无需主动去读取它们。
